## What's Been Created

### Auto-Transition Sidecar Script
Location: `vikunja/auto-transition.sh`

Functionality:
- Runs every 5 minutes (configurable via CHECK_INTERVAL)
- Uses Vikunja API with bearer token authentication
- Processes two types of transitions:
  1. **Blocked tasks**: Checks if all blockers are completed → moves to `state:ready`
  2. **Scheduled tasks**: Checks if `earliest-on:YYYY-MM-DD` date reached → moves to `state:ready`

Label states supported:
- `state:ready` - Ready to work on
- `state:blocked` - Blocked by other task(s)
- `state:scheduled` - Has earliest-on date
- `state:on-halt` - Paused/halted
- `state:in-progress` - Currently being worked on
- `state:completed` - Done

Environment variables:
- `VIKUNJA_API_URL` - API endpoint (default: http://localhost:3456)
- `VIKUNJA_API_TOKEN` - API bearer token (REQUIRED)
- `CHECK_INTERVAL` - Seconds between checks (default: 300)

### GitHub Actions Workflow
Location: `.github/workflows/deploy.yml`

Trigger: push to `vikunja` branch or tags `vikunja-*`

Features:
- Validates DOMAIN_NAME and KUBECONFIG_B64 secrets
- Creates namespace if needed
- Checks for OIDC secret (warns if missing)
- Replaces DOMAIN placeholder in manifests
- Waits for deployment rollout
- Provides summary output

Runner: `k3s-home-runner`

## What Still Needs Implementation

### Add Sidecar to vikunja.yaml
The auto-transition container needs to be added to the Deployment:

```yaml
- name: auto-transition
  image: alpine/curl:latest
  command: ["/scripts/auto-transition.sh"]
  env:
    - name: VIKUNJA_API_URL
      value: "http://localhost:3456"
    - name: VIKUNJA_API_TOKEN
      valueFrom:
        secretKeyRef:
          name: vikunja-api-token
          key: token
    - name: CHECK_INTERVAL
      value: "300"
  volumeMounts:
    - name: auto-transition-script
      mountPath: /scripts
```

Also need:
- ConfigMap for the script
- Secret for API token (vikunja-api-token)

### API Token Generation
Vikunja doesn't have built-in API tokens, options:
1. Use admin user credentials (less secure)
2. Create service account via CLI
3. Use existing user account token (manual setup)

## Testing
1. Generate API token in Vikunja (via UI or CLI)
2. Create secret: `kubectl create secret generic vikunja-api-token --namespace=vikunja --from-literal=token=<token>`
3. Update vikunja.yaml with sidecar
4. Apply manifests
5. Check sidecar logs: `kubectl logs -n vikunja deployment/vikunja -c auto-transition`
